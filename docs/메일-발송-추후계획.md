# 메일 발송 — 운영 방식 및 추후 계획

작성: 2026-05-13

## 현재 운영 방식 (확정) — A. 시스템 계정 발송

- 시스템 설정의 **"메일 발송 GAS URL" 칸은 비워둠**
- 메일 발송 호출이 메인 GAS(현재 운영 중)로 흐름 → 메인 GAS 소유자(시스템) 계정의 Gmail로 발송
- 코드 동작 — `src/components/modals/SendReportEmailModal.js` → `callMailAction(data, mailGasUrl)`
  - `mailGasUrl`이 비어있으면 GAS_URL(메인)로 폴백 (`src/utils/api.js`의 `callMailAction`)
  - 메인 GAS의 `SEND_REPORT_EMAIL` 액션이 처리, `sendReportEmail()` 호출
  - `GmailApp.sendEmail(to, subject, plain, options)` — `options.replyTo`에 작성자 본인 메일 자동 설정
- **받는 이 시점**:
  - 보낸이 표시: `김철수 (MAK-PMS) <mak-pms@회사.co.kr>` 형태
  - 답장 클릭 시 작성자 본인 메일 주소로 전송 (reply-to 라우팅)
- 발송 이력은 메인 GAS 시트의 `MAIL_LOG`에 그대로 기록 (관리자 메뉴 "메일 발송 이력"에서 조회)

## 검토했던 옵션

### B. "Anyone" 배포 후 메일 GAS URL 등록 — **포기**
- 익명 호출 허용 endpoint로 만들면 CORS 통과
- 그러나 GAS가 호출자 ID를 못 얻음 (`Session.getActiveUser().getEmail()` 빈값)
- → 결과적으로 시스템 계정 발송과 동일. 추가 배포만 늘고 의미 없음

### C. Service Account + Domain-Wide Delegation — **추후 계획**
- 본인 Gmail 발송이 정말 필요할 때 진행
- 작업 내용:
  1. Google Cloud Console에서 Service Account 생성
  2. Workspace 관리 콘솔에서 해당 Service Account에 "Gmail 권한 위임" 설정 (Domain-Wide Delegation)
     - 권한 스코프: `https://www.googleapis.com/auth/gmail.send`
  3. GAS 코드 수정:
     - `GmailApp.sendEmail(...)` → Gmail API REST 호출 + Service Account 토큰으로 "user impersonation"
     - 또는 별도 백엔드(Node/Python)에서 처리 후 GAS 분리
  4. 클라이언트(React) 변경 없음 — 그대로 `callMailAction` 호출
- 예상 공수: 0.5~1일 (Workspace Admin 작업 포함)
- 우선순위: **낮음** — 현재 reply-to 방식으로 실용상 큰 불편 없음

## 기록된 시행착오

도메인 제한 메일 GAS URL을 등록하면 발생한 에러:
```
Access to fetch at 'https://script.google.com/a/macros/mak.co.kr/s/.../exec' 
from origin 'http://localhost:3000' has been blocked by CORS policy: 
No 'Access-Control-Allow-Origin' header is present on the requested resource.
api.js:127  POST .../exec net::ERR_FAILED 401 (Unauthorized)
```

- 원인: `/a/macros/도메인/s/...` 형태 URL은 Google이 인증 redirect를 요구 → 브라우저 fetch는 redirect 못 따라감 → 401 + CORS 차단
- 결론: 도메인 제한 endpoint는 fetch 호출 불가 (코드로 못 풀음, Google 정책)

## 재활성화 조건

- 외부 고객 대상 발송 비중이 늘어 발신자 신뢰성이 중요해짐
- 본인 보낸편지함에 발송 기록을 남겨야 한다는 감사/규정 요구

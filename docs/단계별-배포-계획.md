## Phase 1 — 베타 (팀장 + 지정 인원)

### 1.1 대상 선정
- **필수 포함**: 각 도메인(2차전지/반도체/디스플레이)의 팀장 1명씩 + 도메인 시니어 1명
- **추천 포함**: 신규 프로젝트 PM 1~2명 (실무 적용 시나리오)
- **추천 포함**: 운영팀 1명 (관리자 관점 검증)
- **합계**: 5~10명

### 1.2 사전 준비 (개발)
- [ ] 실제 사용자 계정 일괄 생성 (사용자 관리 UI에서)
- [ ] 비밀번호 초기화 정책 (`mustChangePassword: true`)
- [ ] 각 사용자에 권한 부여 (ADMIN / PM / ENGINEER / CUSTOMER + 팀장 토글 + 주간보고 토글)
- [ ] 시드 데이터 정리 (테스트 데이터 삭제 또는 분리)
- [ ] **데이터 백업 자동화** — GAS 시트 매일 자정 자동 백업 (실수로 한 줄 지워도 복구)
- [ ] 사용자 가이드 인쇄/공유 자료 준비 (`MAK-PMS_사용자교육.pptx` 활용)

### 1.3 교육 (1주차)
- **시간**: 90분 × 2회 (오전반/오후반)
- **내용**:
  - 로그인 / 비밀번호 변경
  - 프로젝트 등록 / 셋업 일정 입력 (실습)
  - 회의록·노트 작성 / 첨부
  - 이슈·AS 등록 (실습)
  - 주간 업무 보고 (실습)
  - 알림 센터 / 검색
- **자료**: 슬라이드 (`docs/MAK-PMS_사용자교육-테마적용.pptx`)

### 1.4 피드백 루프 (2~4주차)
- **주간 회고 미팅** (30분, 매주 금요일)
  - 발견 이슈 정리
  - 우선순위 합의
  - 다음 주 핫픽스 계획
- **상시 피드백 채널**: Slack/카톡 채널 또는 시스템 내 "피드백 보내기" 버튼
- **이슈 트래커**: 시스템의 "이슈 관리" 탭 자체 활용 (PRJ-INTERNAL 가상 프로젝트에 등록)

### 1.5 안정화 기준 (Phase 2 진입 조건)
- [ ] 4주간 P0(블로커) 이슈 0건
- [ ] P1(주요) 이슈 모두 해결 또는 완화책 합의

## Phase 3 — 외부 (고객사 접근)

> **이 단계가 가장 큰 변화**. 단순 기능 확장이 아니라 **인프라/보안/법적** 준비가 모두 필요.

### 3.1 인프라 — 무엇이 필요한가

#### A. 호스팅 (어디에 띄울 것인가)

| 옵션 | 설명 | 월 비용 (대략) | 권장도 |
|---|---|---|---|
| **A1. 사내 PC** | 사무실 PC에 서버 띄움 | 무료 | ❌ 외부 서비스용으로 부적합 (정전/재부팅 위험) |
| **A2. 정적 호스팅 (Vercel/Netlify/Cloudflare Pages)** | React build 결과물만 무료 정적 호스팅 | 무료 (Hobby) ~ $20/월 (Pro) | ⭐⭐⭐ **권장 (백엔드는 GAS라 별도 서버 불필요)** |
| **A3. AWS S3 + CloudFront** | AWS 정적 호스팅 + CDN | 월 $5~20 | ⭐ AWS 기존 사용 시 |
| **A4. 자체 VM (AWS EC2 / GCP)** | 가상 서버 임대 | 월 $20~100 | ❌ 오버스펙 (정적 SPA에 VM 불필요) |

**핵심**: MAK-PMS의 백엔드는 **Google Apps Script (GAS)** 이고 이미 Google 인프라에서 무중단 운영됨. **프론트엔드(React build) 만 어딘가 정적 호스팅하면 됨**. 그래서 A2(Vercel/Netlify/Cloudflare Pages) 가 가장 단순 + 저렴.

#### B. 도메인

| 옵션 | 설명 | 연 비용 |
|---|---|---|
| 신규 도메인 구매 (예: `mak-pms.com`) | 가비아/후이즈/Namecheap | 약 ₩15,000~30,000/년 |
| 기존 회사 도메인 서브도메인 (예: `pms.makenton.co.kr`) | IT팀 협의 | 무료 (회사 자산 활용) |

→ **추천**: 회사 서브도메인 사용. 신뢰성 + 브랜드 일관성 + IT 관리 통합

#### C. SSL/HTTPS (인증서) — 무엇이고 왜 필요한가

**SSL = Secure Sockets Layer**. 정확한 현행 표준은 **TLS** (Transport Layer Security)지만 업계에선 관행적으로 "SSL"이라 부름.

**역할 한 줄 요약**: 브라우저(고객) ↔ 서버(우리 시스템) 사이의 모든 데이터를 **암호화** 해서, 중간에 누가 가로채도 못 읽게 만드는 것.

**왜 필요한가**:
1. **데이터 보호**: 비밀번호, 회의록, AS 사진 등이 평문으로 인터넷을 떠다니면 누구나 도청 가능. SSL로 암호화하면 가로채도 의미 없는 바이트.
2. **신원 인증**: 사용자가 보는 `mak-pms.회사.co.kr`이 진짜 우리 회사 서버인지 보장 (가짜 사이트로 비번 빼가는 피싱 차단).
3. **브라우저 정책**: 최신 브라우저(Chrome/Edge/Safari)는 HTTPS 없으면 **"안전하지 않음"** 빨간 경고. 외부 고객이 보면 즉시 이탈. 일부 기능(로그인 폼, 파일 다운로드)은 아예 동작 안 함.
4. **검색/SEO**: Google이 HTTPS 사이트를 우선 노출.

**작동 원리 (간단)**:
```
[고객 브라우저]  ──"안녕, 너 진짜 mak-pms.회사.co.kr 맞아?"──>  [서버]
[고객 브라우저]  <──"응, 여기 인증기관(CA)이 서명한 인증서야"──  [서버]
[고객 브라우저] 인증서 검증 → 신뢰할 수 있는 CA(예: Let's Encrypt) 서명이면 OK
                ↓
        암호화된 통신 시작 (대칭키 교환 후)
        이후 모든 요청/응답이 도청 불가
```

**인증서를 어디서 받나**:
| 발급기관 | 비용 | 갱신 | 우리 케이스 |
|---|---|---|---|
| **Let's Encrypt** (CA 비영리) | **무료** | 90일마다 자동 | ⭐⭐⭐ 권장 |
| Cloudflare | 무료 | 자동 | ⭐⭐⭐ Pages 사용 시 자동 포함 |
| 상용 (DigiCert, Sectigo 등) | 연 ₩50K~수백만원 | 1~2년마다 수동 | 금융 등 특수 케이스만 |

**우리 상황에서 SSL 셋업은 거의 자동**:
- Vercel/Netlify/Cloudflare Pages에 도메인 등록 → **자동으로 Let's Encrypt 인증서 발급 + 90일 자동 갱신**
- 별도 작업/비용 0원
- 셋업 시간: 도메인 DNS 설정만 하면 5분 내 활성화

**자체 서버 띄울 경우** (참고):
- `certbot` (Let's Encrypt 공식 도구) 설치 → `certbot --nginx -d mak-pms.회사.co.kr` 한 줄로 발급
- 90일 만료 전 자동 갱신 cron 설정
- 비용 무료지만 운영 책임이 우리에게 있음

**검증 방법** (셋업 후):
- 브라우저 주소창 자물쇠 아이콘 → "보안 연결" 표시 확인
- `https://www.ssllabs.com/ssltest/` 에 도메인 입력 → A 등급 이상이면 안전
- 만료일 모니터링 (Vercel 등은 자동, 자체 서버는 알림 설정)

#### D. 회사 인터넷 회선 / 방화벽
- **사내 PC 호스팅 안 함**: 외부 접속용 회선 / 방화벽 / 고정 IP 셋업 불필요
- **사내 자료 접근만 필요**: VPN 또는 IP 화이트리스트 (다음 단계 보안 참고)

#### E. 백업 / 복구
- GAS 시트 자동 백업 → Drive에 일별 사본 (스케줄 트리거)
- 1주일치 보관 + 월 1회 장기 보관

#### F. 모니터링
- **Vercel/Netlify**: 기본 분석 (방문 수, 응답 시간)
- **Google Workspace 감사 로그**: 사용자 활동, Drive 접근
- 추가 도구 (Sentry, LogRocket 등)는 외부 서비스 가입 후 SDK 삽입 — 선택

#### G. 비용 추산 (Phase 3 진입 후)
| 항목 | 월 비용 |
|---|---|
| 도메인 | ~₩2,500 (연 비용 ₩30K / 12) |
| 정적 호스팅 (Vercel Pro) | ₩25,000 (트래픽 많을 때만, 처음엔 무료) |
| GAS (Workspace 기본 포함) | ₩0 |
| Drive 용량 (필요 시 추가) | ₩2,000/사용자 (Business Plan 일부) |
| **합계** | **월 ₩30~50K 수준** |

→ 본격 외부 노출 인프라치고는 매우 저렴 (GAS 백엔드 덕분).

### 3.2 보안 — 무엇이 필요한가 (상세)

> 우선순위: 🔴 **필수** (없으면 외부 노출 금지) / 🟡 **강력 권장** / 🟢 **선택**

#### 🔴 S1. HTTPS 강제
- **현재**: 사내 환경에선 HTTP도 OK
- **목표**: 모든 트래픽 HTTPS, HTTP 접속 시 자동 HTTPS 리다이렉트
- **구체 작업**:
  - Vercel/Cloudflare 사용 시 → 자동 (별도 작업 없음)
  - 자체 호스팅 시 → nginx `return 301 https://$host$request_uri;`
  - **HSTS 헤더** 추가 (`Strict-Transport-Security: max-age=63072000`): 한번 HTTPS 접속한 브라우저는 향후 HTTP 시도조차 안 함
- **공수**: 호스팅 자동이면 0, 자체면 1h

#### 🔴 S2. 비밀번호 정책 강화
- **현재**: SHA-256 + 솔트 (해시는 OK), 자유 입력 허용
- **목표**:
  - 최소 8자, 영문 대소문자 + 숫자 1자 이상 (특수문자는 권장)
  - 사전 단어/연속 숫자 금지 (`12345678`, `password` 차단)
  - 비밀번호 변경 강제: 첫 로그인 / 90일 주기
  - 이전 3개 비밀번호 재사용 금지
- **구체 작업**:
  - `src/utils/auth.js`에 검증 함수 추가
  - 로그인 모달에 강도 미터 표시 (약함/보통/강함)
  - `users` 시트에 `passwordHistory: string[]`(최근 3개 해시), `passwordChangedAt` 컬럼 추가
- **공수**: 0.5d

#### 🔴 S3. 세션 관리
- **현재**: localStorage에 사용자 정보 보관 → 만료 없음 (영구 로그인)
- **위험**: PC 도난·공용 PC에서 로그인 후 미로그아웃 → 누구나 접근
- **목표**:
  - JWT 토큰 + 만료 시간 (예: 12시간)
  - 비활동 30분 후 자동 로그아웃 경고
  - "내 PC에서만 로그인" / "공용 PC" 옵션 (후자는 만료 짧게)
  - 로그아웃 시 토큰 무효화 (블랙리스트 시트)
- **구체 작업**:
  - GAS에 토큰 발급/검증 함수 추가
  - 클라이언트 `useEffect`로 활동 감지 + 자동 로그아웃
- **공수**: 1d

#### 🔴 S4. CORS / Origin 검증
- **현재**: GAS Web App은 누구나 호출 가능 (URL만 알면 외부에서 임의 호출 가능)
- **위험**: 누군가 GAS URL 알면 데이터 조회/수정 API 직접 호출
- **목표**: 정해진 프론트 도메인(`mak-pms.회사.co.kr`)에서만 요청 허용
- **구체 작업** (GAS):
  ```js
  function doPost(e) {
    var origin = e.parameter.origin || (e.postData && JSON.parse(e.postData.contents).origin);
    var ALLOWED = ['https://mak-pms.회사.co.kr', 'http://localhost:3000']; // 개발용
    if (ALLOWED.indexOf(origin) === -1) {
      return ContentService.createTextOutput(JSON.stringify({ error: 'Forbidden origin' }));
    }
    // 기존 로직
  }
  ```
  - 클라이언트 `src/utils/api.js`에서 `origin: window.location.origin` 본문에 포함
- **주의**: GAS는 HTTP `Origin` 헤더를 자동으로 받지 못함 → 본문에 명시적으로 넣어야 함
- **공수**: 0.5d

#### 🔴 S5. 권한 격리 (Authorization)
- **현재**: role 기반 (ADMIN/PM/ENGINEER/CUSTOMER)이 있지만 **데이터 행 단위 검증은 클라이언트에서만** 수행 → 악의적 사용자가 다른 고객 데이터 직접 요청 가능
- **목표**: **모든 데이터 응답을 GAS에서 사용자 권한별로 필터링** (서버 사이드 검증)
- **구체 작업**:
  - GAS `loadAll()` 응답을 사용자별 가공:
    - CUSTOMER: 본인 회사 프로젝트만 (`projects.filter(p => p.customer === user.customer)`)
    - ENGINEER: 배정된 프로젝트만 (`assignedEngineerIds.includes(user.engineerId)`)
    - PM: 본인 담당 프로젝트만 (`manager === user.name`)
    - ADMIN: 전체
  - 각 mutation 핸들러(`UPDATE_PROJECT` 등)에 권한 체크 추가
- **공수**: 1.5d (모든 API 점검 필요)
- **테스트**: 모의 고객 계정으로 다른 고객 데이터 접근 시도 → 차단 확인

#### 🔴 S6. 입력 검증 (Injection 방어)
- **현재**: 사용자 입력 그대로 시트/Drive에 저장 + 그대로 화면 렌더
- **위험**:
  - XSS: 사용자가 `<script>...</script>` 입력 시 다른 사용자 화면에서 실행될 수 있음 (React는 기본 escape하지만 `dangerouslySetInnerHTML` 쓰면 위험)
  - 시트 수식 주입: `=IMPORTRANGE("악성URL")` 같은 셀 수식이 자동 실행될 수 있음 (Google Sheets는 `=` 시작 셀을 수식으로 해석)
  - 파일명 path traversal: `../../../etc/passwd` 같은 파일명
- **목표**:
  - 모든 텍스트 입력 `'=' '+'`로 시작하면 앞에 `'` 추가 (시트 수식 차단)
  - 파일명에서 `/`, `\`, `..` 제거
  - React 컴포넌트 어디서도 `dangerouslySetInnerHTML` 사용 안 함 (현재 점검 필요)
- **구체 작업**:
  - `src/utils/sanitize.js` 신규: `sanitizeForSheet(text)`, `sanitizeFileName(name)`
  - GAS `appendRow` 호출 전 sanitize
  - 코드 grep으로 `dangerouslySetInnerHTML` 전수 검토
- **공수**: 0.5d

#### 🟡 S7. Rate Limiting
- **현재**: 없음
- **위험**: 자동화 봇이 로그인 무차별 시도, 또는 다운로드 폭주로 GAS 할당량 소진
- **목표**:
  - 로그인: IP당 5회/분 실패 시 15분 차단
  - API 호출: 사용자당 60회/분 (조회), 10회/분 (쓰기)
- **구체 작업** (GAS):
  ```js
  function checkRateLimit(userId, action) {
    var key = userId + ':' + action;
    var cache = CacheService.getScriptCache();
    var count = parseInt(cache.get(key) || '0', 10);
    if (count > LIMITS[action]) throw new Error('Too many requests');
    cache.put(key, String(count + 1), 60);
  }
  ```
- **공수**: 0.5d

#### 🟡 S8. 로그인 보호 (CAPTCHA / 2FA)
- **현재**: 없음
- **목표**:
  - 3회 실패 시 reCAPTCHA 활성화 (Google v3, 사용자 마찰 적음)
  - **2단계 인증 (선택)**: 핸드폰 SMS 또는 OTP 앱 (TOTP)
- **구체 작업**:
  - reCAPTCHA v3 통합 (~2h)
  - 2FA는 `otplib` + QR 코드 생성, 사용자별 secret 시트 보관 (~1d)
- **공수**: reCAPTCHA 0.5d, 2FA 추가 시 +1d

#### 🟡 S9. 감사 로그 (Audit Trail)
- **현재**: 프로젝트별 `activityLog`만 있음 (생성/수정/완료 단위)
- **목표**:

- **구체 작업**:

## 부록. Phase 3 인프라 셋업 — 최소 비용 시나리오

가장 단순/저렴한 외부 노출 구성:

```
[고객 브라우저]
        │
        ▼ HTTPS
[mak-pms.회사도메인.co.kr]  ← 회사 IT팀에 CNAME 추가 요청
        │
        ▼
[Vercel/Cloudflare Pages]  ← React build 자동 배포 (GitHub 연동)
        │
        ▼ HTTPS
[Google Apps Script Web App]  ← 기존 GAS 백엔드 (변경 없음)
        │
        ▼
[Google Sheets + Drive]  ← 기존 데이터 저장소 (변경 없음)
```

- **총 추가 비용**: 도메인 연 ₩30K + 무료 호스팅 = **연 ₩30,000 미만** (트래픽 적을 때)
- **셋업 시간**: 회사 IT 도메인 등록 후 호스팅 연결까지 약 **반나절**
- **운영 비용**: 추가 없음 (GAS 무료 한도 내, Drive 기존 사용)

→ 외부 노출이라고 거창한 인프라가 필요 없음. 핵심은 **보안/법적 준비**가 진짜 시간 듦.

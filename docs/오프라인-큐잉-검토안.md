# 오프라인 큐잉 (Offline Queue) 도입 검토안

> **목적**: 인터넷 연결이 끊긴 환경(공장 내부, 지하, 이동 중 등)에서 사용자가 작성한 데이터를 잃지 않고, 연결 복구 시 자동으로 서버에 반영되도록 한다.
>
> **상태**: 검토안 (의사결정 전). 업무 협의 후 범위 확정 → 일정 산정.
>
> **작성일**: 2026-05-13

---

## 1. 배경 / 문제 정의

### 현재 동작
- 입력 폼 데이터는 **브라우저 메모리(React state)** 에만 존재.
- 저장 버튼 클릭 시 GAS 백엔드(`fetch`)로 즉시 POST → 성공 응답 받으면 클라이언트 state 갱신.
- 네트워크 실패 시: `fetch`가 reject → 에러 토스트 표시 → 사용자가 저장 버튼 재클릭해야 함.

### 실제 발생 가능한 시나리오
| 상황 | 결과 |
|---|---|
| 공장 안에서 AS 접수 작성 (와이파이 약함) | 저장 시도 시 실패 → 사용자가 작성 내용 보존 위해 화면 떠나지 못함 |
| 작성 중 새로고침 / 탭 닫기 | **데이터 100% 손실** |
| 출장 중 모바일 데이터 끊김 | 사진 첨부 업로드 중단 시 처음부터 다시 |
| 동시 작업자 다수, 한 명이 GAS 재배포 | 1~2초 짜리 요청 모두 실패 (오프라인과 동일한 효과) |

### localStorage만으로 충분한가?
**부족함**. localStorage는 5MB 한계라 사진 첨부 큐잉이 불가능. 또한 동기 API라 큰 데이터 쓰기 시 UI 멈춤.

→ **IndexedDB** 필수.

## 2. 요구사항

### Functional
- [F1] 오프라인 상태에서 "저장" 클릭 시 데이터를 로컬 큐에 적재한다.
- [F2] 온라인 복귀 시 자동으로 큐를 순서대로 비우며 서버로 전송한다.
- [F3] 첨부 파일(사진/문서)도 큐잉되어야 한다 (base64로 보관).
- [F4] 사용자에게 현재 큐 상태(대기 N건, 동기화 중, 완료, 실패)를 시각적으로 표시한다.
- [F5] 큐 항목 재전송 시 **중복 등록을 막아야 한다** (멱등성).
- [F6] 큐 전송 실패 시 재시도 정책이 명확해야 한다 (지수 백오프 / 수동 재시도 / 폐기 선택).

### Non-functional
- [NF1] 오프라인 작성 → 온라인 복귀 → 자동 전송까지 사용자가 별도 액션 필요 없음.
- [NF2] 큐 데이터는 브라우저를 닫아도 보존되어야 한다 (IndexedDB).
- [NF3] 큐 크기가 커져도 (사진 10장 × 5건) UI가 멈추지 않아야 한다 (비동기 + 청크).
- [NF4] 같은 사용자가 다른 디바이스로 옮기면 큐는 따라가지 않는다 (로컬 전용).

### Out of scope (이번 범위에서 빼는 것)
- 다중 디바이스 간 큐 동기화 (PC에서 작성 → 모바일에서 마저)
- 오프라인 중 다른 사용자의 변경사항 미리보기 (실시간 협업)
- 오프라인 중 데이터 **수정**, **삭제** (신규 등록만 우선)

## 3. 범위 옵션

| 옵션 | 포함 작업 | 예상 공수 | 권장도 |
|---|---|---|---|
| **A** | AS 접수 + 사진 첨부만 | 1일 | ⭐ 현장 우선 |
| **B** | AS + 노트 + 댓글/응답 | 1.5~2일 | ⭐⭐ 권장 |
| **C** | A/B + 이슈 등록 + 셋업 완료 사인 + 출장 등록 | 2.5~3일 | 풀 커버 |
| **D** | 모든 쓰기 동작 (수정/삭제 포함) | 5일+ | 오버엔지니어링 위험 |

**제안**: B (AS + 노트 + 댓글) — 실제 현장 시나리오 80% 커버, 충돌 가능성 낮음.

## 4. 기술 설계

### 4.1 큐 스토리지 구조 (IndexedDB)

```
DB: eq_pms_offline_queue
└── store: outbox
    └── record:
        {
          id: 'uuid-v4',              // 클라이언트 생성 UUID
          createdAt: 1715600000000,
          op: 'AS_ADD' | 'NOTE_ADD' | 'AS_COMMENT' | ...,
          projectId: 'PRJ-xxx',
          payload: { ... },            // 서버로 보낼 데이터
          files: [                     // 첨부
            { name, type, size, base64 }
          ],
          status: 'pending' | 'sending' | 'failed',
          attempts: 0,
          lastError: null,
          lastTriedAt: null
        }
```

### 4.2 핵심 모듈 (신규)

```
src/lib/offline/
├── db.js              # IndexedDB wrapper (open/get/put/delete/all)
├── queue.js           # enqueue, dequeue, peek, retry
├── sync.js            # online/offline 이벤트, replay 로직
└── useOfflineQueue.js # React hook (큐 상태 + 액션 노출)
```

### 4.3 데이터 흐름

```
[사용자 저장 클릭]
        │
        ▼
  ┌──────────────┐
  │ navigator.   │
  │   onLine ?   │
  └──────┬───────┘
         │
   ┌─────┴─────┐
   │           │
  YES          NO
   │           │
   ▼           ▼
 [GAS 호출]  [IndexedDB outbox에 적재]
   │           │
   ▼           ▼
 [성공] ←─ [online 이벤트 발생 시]
   │           │
   ▼           ▼
 [state 갱신]  [큐 순회하며 1건씩 GAS 전송]
                │
                ▼
              [성공한 항목 삭제, state 갱신]
              [실패 시 attempts++ → 백오프 후 재시도]
```

### 4.4 멱등성 (중복 방지)

**핵심**: 같은 클라이언트 UUID가 두 번 전송돼도 GAS는 한 건만 기록.

**클라이언트**:
```js
const id = crypto.randomUUID();  // 큐 적재 시 생성
payload.clientId = id;            // GAS로 전송
```

**GAS 백엔드** (`AS_ADD` 예시):
```js
function doPost(e) {
  const data = JSON.parse(e.postData.contents);
  if (data.action === 'AS_ADD') {
    const sheet = SpreadsheetApp.getActive().getSheetByName('AS_LOG');
    const rows = sheet.getDataRange().getValues();
    // clientId 컬럼이 이미 존재하면 중복 → 무시
    if (rows.some(r => r[CLIENT_ID_COL] === data.clientId)) {
      return ContentService.createTextOutput(JSON.stringify({ ok: true, deduped: true }));
    }
    // 신규 등록
    sheet.appendRow([...]);
  }
}
```

→ 네트워크 끊김으로 클라이언트가 응답 못 받고 재전송해도 안전.

### 4.5 충돌 처리 (오프라인 작성 vs 다른 사용자 동시 작성)

**시나리오**: A가 오프라인에서 PRJ-001 AS를 등록. 그동안 B가 같은 PRJ-001에 AS를 등록. A가 온라인 복귀 시 두 AS 모두 살아남아야 함.

| 옵션 | 동작 | 복잡도 |
|---|---|---|
| **Last-write-wins** (권장) | 둘 다 별도 레코드로 적재. AS는 별도 ID 가지므로 충돌 없음. | ★ |
| 머지 UI | 같은 ID의 수정이 충돌하면 사용자에게 비교 화면 | ★★★★ |
| 신규만 큐잉 | 신규 등록만 오프라인 가능, 수정/삭제는 온라인 필수 | ★★ (권장 대안) |

**결론**: B 범위(신규 등록 중심)에선 충돌 자체가 거의 발생하지 않음 → Last-write-wins로 충분.

### 4.6 UI 표시

**상단 글로벌 배너** (offline 상태 시):
```
┌────────────────────────────────────────────────┐
│ ⚠ 오프라인 — 작성 내용은 자동 저장됩니다 (대기 3건) │
└────────────────────────────────────────────────┘
```

**복귀 직후**:
```
┌────────────────────────────────────────────────┐
│ ↻ 연결 복구 — 대기 3건 전송 중... (1/3)         │
└────────────────────────────────────────────────┘
```

**완료**:
```
┌────────────────────────────────────────────────┐
│ ✓ 동기화 완료 — 3건 모두 반영됨                  │
└────────────────────────────────────────────────┘ (5초 후 자동 닫힘)
```

**실패 항목 별도 패널** (사이드바 하단 또는 알림 센터):
- 큐 항목 클릭 → 페이로드 미리보기 + "재시도" / "삭제" 버튼

## 5. GAS 백엔드 변경사항

### 신규 컬럼 추가
- AS_LOG, NOTES, COMMENTS 등 각 시트에 **`clientId`** 컬럼 추가 (선택적, 없으면 일반 등록).

### 코드 변경
- 각 `doPost` 핸들러에 멱등성 체크 추가 (UUID 기반).
- 약 50~80줄 추가, 기존 로직 영향 없음.

### 배포
- 기존 GAS와 동일하게 새 버전 배포 (URL 유지).
- 구버전 클라이언트는 `clientId` 없이 보내도 정상 작동 (하위 호환).

## 7. 리스크 / 제약

### 기술 리스크
| 항목 | 영향 | 완화 방안 |
|---|---|---|
| IndexedDB 용량 초과 (사진 大용량 多건) | 데이터 손실 | 큐 50건 + 사진 10MB 초과 시 경고 + 사용자에게 정리 유도 |
| 브라우저 캐시 정리로 큐 삭제 | 데이터 손실 | "오프라인 대기 중에는 캐시 삭제 금지" 경고 + 영구 저장 요청 (`navigator.storage.persist()`) |
| 사파리/iOS 사파리 IndexedDB 버그 | 일부 환경 미동작 | 폴백: localStorage(작은 텍스트만) + 사용자 알림 |
| 동일 사용자 다중 탭 충돌 | 같은 큐 항목 두 번 전송 | BroadcastChannel로 탭 간 리더 선출 (1탭만 sync 실행) |

### 운영 리스크
| 항목 | 완화 |
|---|---|
| 사용자가 오프라인 모드 인지 못함 | 상단 배너 + 빨간색 시각 강조 |
| 큐가 쌓인 채 사용자가 퇴근 → 다음날 출근하면? | 자동 재시도 + "어제 작성한 N건 전송 완료" 알림 |
| 충돌(같은 항목 수정) 발생 | 옵션 B는 신규만 큐잉 → 거의 없음 |

### 비기능 제약
- **모바일 PWA화는 별개 작업**. 이번 범위에서는 PC/모바일 브라우저만 지원.
- **푸시 알림으로 동기화 진행 알림**은 제외 (Service Worker 도입 필요 → 다음 단계).

## 8. 의사결정 필요 항목

업무 협의에서 결정해야 할 사항:

1. **범위 선택**: 옵션 A / B / C 중 어느 것?
2. **충돌 정책**: Last-write-wins로 충분한가, 충돌 시 사용자 확인 UI가 필요한가?
3. **GAS 변경 승인**: 멱등성 컬럼 추가 + 재배포 OK인가?
4. **실패 큐 폐기 권한**: 며칠 이상 실패한 큐 항목은 자동 폐기할까, 영구 보관할까?
5. **모바일 우선순위**: 현재 사용자가 모바일 브라우저로도 작성하는가? (Service Worker 도입 필요 여부)
6. **테스트 환경**: 본격 배포 전 1주일 테스트 사용자(2~3명)에게 베타 제공할까?

## 부록 B. 참고 — 외부 라이브러리 사용 여부

- **IndexedDB wrapper**: 자체 구현 권장 (간단함). `idb` (Jake Archibald) 라이브러리 대안 가능 (~3KB gzip).
- **UUID**: `crypto.randomUUID()` 네이티브 사용 (Chrome 92+, Safari 15.4+). 폴백: 간단한 `Math.random().toString(36)` 조합.
- **상태 관리**: 기존 React Context로 충분. Zustand 등 추가 도입 불필요.
